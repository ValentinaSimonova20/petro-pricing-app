# Шаг 2 / День 3: первые расчёты

Документ — план реализации **основной цепочки расчёта цены** по паспорту кейса из презентации СИБУР × ЦУ и текущему состоянию репозитория.

**Образ результата дня (из презентации):**  
строка прогноза → формула → подстановка чисел → итоговая цена и расшифровка на экране.

**Фокус дня:** получить первые посчитанные цены; ошибка в одной строке не должна останавливать весь расчёт.

## 1. Цель и границы Дня 3

### Цель
Реализовать end-to-end расчёт для строк прогноза (`DemandForecast` / `ssp`), используя уже загруженные справочники:

1. подобрать формулу(ы) по области применения;
2. подставить значения термов (котировки, курсы, константы);
3. вычислить цену по тексту формулы;
4. показать результат и расшифровку в UI;
5. явно учитывать **нестандартные ситуации** (нет формулы, нет котировки, битое выражение и т.д.) и выводить **метрики** по ним.

### Входит в День 3
- Кнопка «Запустить расчёт» (параметры: период / фильтр — минимум).
- Backend pipeline: match → resolve → evaluate → persist/return.
- Вкладка/экран **«Результаты расчёта»** + **расшифровка строки**.
- Панель **метрик / KPI расчёта** (успех, ошибки, невозможность расчёта).
- Устойчивость: сбой одной строки → статус по строке, остальные считаются дальше.

### Не входит в День 3 (День 4 по презентации)
- Ручные правки цены и смена формулы пользователем (достаточно **показать** несколько кандидатов).
- Выгрузка Excel.
- Полный сценарий приёмки + CI/CD + README к защите.
- Сверка с эталоном Pricing на полном наборе (заложить структуру метрики «совпадение ±3%», но эталон может появиться позже).

## 2. Целевая цепочка (логический пайплайн)

```text
DemandForecast (ssp)
        │
        ▼
┌───────────────────────┐
│ 1. Подбор формулы     │  client_id, материал/группа M, период, contract,
│    FormulaMatcher     │  Неактивна = нет, окно valid_from/valid_to
└───────────┬───────────┘
            │ 0 / 1 / N кандидатов
            ▼
┌───────────────────────┐
│ 2. Выбор default      │  N → формула с max(Дата создания);
│                       │  все кандидаты сохранить для расшифровки
└───────────┬───────────┘
            ▼
┌───────────────────────┐
│ 3. Компоненты         │  formula_components по Ключ формулы
│    FormulaComponent   │  тип терма → term_types
└───────────┬───────────┘
            ▼
┌───────────────────────┐
│ 4. Резолв значений    │  константа / котировка / курс
│    ValueResolver      │  котировка: Факт → ОФ → ППР
└───────────┬───────────┘
            ▼
┌───────────────────────┐
│ 5. Вычисление         │  подстановка в текст «Формула»
│    ExpressionEngine   │  цена + валюта документа
└───────────┬───────────┘
            ▼
┌───────────────────────┐
│ 6. Результат + UI     │  таблица, расшифровка, метрики статусов
└───────────────────────┘
```

Имена Excel-колонок — как в `data-model.md`; в Java-слое уже есть нормализованные поля сущностей (`formulaKey`, `mtrNsiCode`, `clientId` и т.д.).

## 3. План работ по этапам

### Этап A. Модель результата и статусы (сначала контракт)

Зафиксировать DTO / сущность результата **до** UI, чтобы все ветки ошибок имели единый код.

**Рекомендуемые поля результата по строке:**

| Поле | Назначение |
| --- | --- |
| `forecastId` / `rowId` + `period` | ссылка на строку прогноза |
| `status` | итоговый статус расчёта строки (см. §4) |
| `statusReason` | человекочитаемое объяснение |
| `selectedFormulaKey` | формула, по которой считали (или null) |
| `candidateFormulaKeys` | все подходящие ключи |
| `price` | итоговая цена или null |
| `currency` | валюта (из формулы / строки) |
| `extendedValidity` | true, если сработало учебное «продление» `valid_to` |
| `components[]` | расшифровка термов: имя переменной, тип, значение, источник, дата, предупреждение |
| `warnings[]` | мягкие замечания (не блокируют цену) |

**Хранение (на выбор команды):**
- in-memory / сессия после POST `/calculate` — быстрее для MVP дня;
- или таблица Liquibase `calculation_result` (+ опционально `calculation_component`) — лучше для F5 и демо.

Рекомендация: **сохранять в H2** (без FK), чтобы перезагрузка страницы не обнуляла результаты, пока живёт процесс.

### Этап B. Подбор формулы (`FormulaMatcher`)

Правила области применения (из презентации + `data-model.md`):

1. **Клиент:** `formulas.client_id` = `ssp.client_id`.
2. **Материал / группа M:**
   - если у формулы задан `Материал` (`material_code`) → равен `ssp.mtr_nsi_code` (с приведением типов/строк);
   - если задана `Подгруппа материалов` (`material_group_m` / `hname`) → материал строки входит в группу через `material_groups` (`code_nsi` = `mtr_nsi_code`);
   - если оба пусты — уточнить правило (для MVP: не матчить «на всё подряд»; логировать как `FORMULA_SCOPE_AMBIGUOUS` или исключать).
3. **Период:** `ssp.period` попадает в окно действия формулы (`valid_from` … `valid_to`).
4. **Неактивные:** `Неактивна` / `inactive` = `X` → **не участвуют** в подборе.
5. **Просроченные (учебное правило MVP):** если формула активна, но `valid_to` < периода строки — **продлевать** до периода строки и помечать `extendedValidity = true` («подбор с продлением срока действия»).
6. **Тип сделки (`contract`):** `Formula` / `Spot` — учесть в матчере (как минимум не ломать строки Spot без формулы; зафиксировать явное правило: Spot → отдельный статус / без формульного расчёта, если в данных нет подходящей записи).
7. **Несколько формул:** вернуть **все** кандидаты; default для авторасчёта — с **самой поздней `Дата создания` / `createdOn`**.

Выход матчера: `MatchResult` (candidates, selected, flags).

### Этап C. Резолв компонентов (`ValueResolver`)

Для выбранной формулы загрузить `formula_components` по `formula_key`.

По `term_types.category` / `type_code`:

| Категория | Источник значения |
| --- | --- |
| константа / фикс. терм | поле `Значение` / `valueAmount` |
| котировка | `quote_mapping` (имя + происхождение) → `quotes` по `quote_code` / `dl_id` |
| курс | `currency_rates` по валюте и дате / правилам периода |

**Котировки — приоритет типа публикации:**  
`Факт` → `ОФ` → `ППР` (на дату/период строки прогноза).

**Курсы:** значение на `calday` (или по правилу периода из компонента — для MVP можно упростить до даты периода / ближайшего дня с явной пометкой в расшифровке).

Если значение не найдено — статус строки `MISSING_QUOTE` / `MISSING_FX` (см. §4), цену не выдумывать.

### Этап D. Вычисление выражения (`ExpressionEngine`)

1. Взять текст `Формула` / `formulaText`.
2. Подставить значения переменных (`Имя переменной` / `variableName`).
3. Вычислить на **сервере** (не в браузере): простой безопасный evaluator (например, библиотека math-expr / свой парсер арифметики). **Не** использовать произвольный `eval` скриптов.
4. Обработать синтаксические ошибки → `EXPRESSION_ERROR`.
5. Зафиксировать валюту результата (`ВалютаДокумента` / `currencyDocument`).

Для Дня 3 достаточно арифметики `+ - * / ( )` и числовых подстановок; сложные «если…то…» — по возможности упростить или помечать `UNSUPPORTED_EXPRESSION`.

### Этап E. Оркестратор и API

- `CalculationService.calculateAll(params)` / `calculate(period, filters)`.
- REST или form POST: `POST /calculate` (HTMX-friendly, как upload).
- Идемпотентность на уровне дня: перед прогоном очищать предыдущие результаты расчёта (аналог `deleteAll` для таблицы результатов).
- Прогресс опционален; обязателен итоговый лог: сколько строк OK / FAIL.

### Этап F. UI

Расширить `index.html` (или отдельный экран):

1. Блок параметров + кнопка **«Посчитать»**.
2. Вкладка **«Результаты расчёта»**:
   - ключевые поля строки прогноза;
   - статус (цвет/бейдж);
   - выбранная формула;
   - цена / валюта;
   - признак продления срока;
   - число кандидатов формул.
3. **Расшифровка** (модалка / раскрытие строки / отдельный HTMX-фрагмент):
   - список кандидатов формул;
   - текст формулы;
   - таблица термов: переменная, тип, значение, источник (котировка/курс/константа), дата, примечание;
   - причина ошибки, если цены нет.
4. Панель **метрик** (см. §5) — всегда видна после расчёта.

Правило UX из презентации: если нет формулы или нет котировки — **сообщение по конкретной строке**, а не «молча» неверная цена.

## 4. Нестандартные ситуации и статусы

Каждая строка получает **один итоговый статус** (жёсткий исход) и опционально список `warnings`.

| Код статуса | Когда | Цена | Поведение |
| --- | --- | --- | --- |
| `OK` | формула найдена, все нужные значения резолвятся, выражение посчитано | есть | обычная строка |
| `OK_EXTENDED_VALIDITY` | как `OK`, но сработало продление `valid_to` | есть | warning в UI |
| `OK_MULTI_FORMULA` | как `OK`, но кандидатов > 1 | есть | в расшифровке список; default = latest created |
| `NO_FORMULA` | ни одна формула не подошла | null | явная причина |
| `SPOT_NO_FORMULA` | `contract = Spot` и формульный расчёт не применим / нет записи | null | отделить от общей «нет формулы» |
| `INACTIVE_ONLY` | были бы кандидаты, но все неактивны | null | редко, для прозрачности |
| `MISSING_COMPONENTS` | у формулы нет строк в `formula_components` | null | |
| `MISSING_QUOTE` | нет котировки на дату / после приоритета Факт→ОФ→ППР | null | указать имя котировки |
| `MISSING_QUOTE_MAPPING` | имя котировки не стыкуется с `quote_mapping` / пустой `dl_id` | null | |
| `MISSING_FX` | нет курса на нужную дату/валюту | null | |
| `MISSING_CONSTANT` | ожидалась константа, `Значение` пусто | null | |
| `AMBIGUOUS_MATERIAL_SCOPE` | конфликт/дыра в связи материал ↔ группа | null | |
| `EXPRESSION_ERROR` | синтаксис / деление на 0 / неизвестная переменная | null | текст ошибки парсера |
| `UNSUPPORTED_EXPRESSION` | конструкция формулы вне MVP-движка | null | |
| `DATA_INCONSISTENT` | битые даты, пустой `client_id`, пустой период и т.п. | null | |
| `INTERNAL_ERROR` | непредвиденное исключение по строке | null | лог + короткий reason; расчёт остальных продолжается |

**Обязательные инварианты:**
- ошибка одной строки **не прерывает** пакетный расчёт;
- цена выставляется **только** при успешном вычислении;
- в UI статусы и причины видны без устных пояснений.

## 5. Метрики (KPI экрана после расчёта)

Метрики считаются **по результатам текущего прогона** (N = число обработанных строк прогноза, с учётом фильтра периода).

### 5.1. Основные (из презентации / приёмки)

| Метрика | Формула | Заметка для Дня 3 |
| --- | --- | --- |
| Доля успешно посчитанных | `(OK + OK_* ) / N` | целевые пороги приёмки: ≥90% отлично, 75–89% хорошо |
| Доля с найденной формулой | `(строки с ≥1 кандидатом) / N` | отдельно от «цена посчитана» |
| Доля совпадения с эталоном ±3% | при наличии эталона | заложить поле/метрику; заполнять, когда появится эталонный файл |

### 5.2. Метрики ошибок и невозможности расчёта (обязательны в День 3)

Показывать **счётчики и доли** по статусам из §4, как минимум:

| Метрика | Зачем |
| --- | --- |
| `count(NO_FORMULA)` / доля | дыры в каталоге / области применения |
| `count(SPOT_NO_FORMULA)` | спотовые строки без формульного пути |
| `count(MISSING_QUOTE)` + `MISSING_QUOTE_MAPPING` | качество котировок и маппинга |
| `count(MISSING_FX)` | дыры в курсах |
| `count(EXPRESSION_ERROR)` + `UNSUPPORTED_EXPRESSION` | качество парсера / сложность формул |
| `count(MISSING_COMPONENTS)` | битая связка формула↔компоненты |
| `count(INTERNAL_ERROR)` | регрессии сервиса |
| Доля строк с warnings (продление срока, multi-formula) | прозрачность «серых» зон |
| Доля multi-formula | насколько часто default «latest created» критичен |
| Top-N причин (`statusReason`) | для демо и разбора с экспертом |

### 5.3. Операционные

| Метрика | Назначение |
| --- | --- |
| Время расчёта (ms / сек) | демо «считается быстро» |
| Число строк в прогоне | сверка с фильтром |
| Число строк с расшифровкой, доступной в UI | для чекпоинта |

Ориентиры приёмки из презентации (полный сценарий — скорее День 4–5): 500 строк, расшифровка у 50 строк, KPI на экране видны.

## 8. Критерии готовности Дня 3

Считаем день выполненным, если:

1. После загрузки справочников (Шаг 1) пользователь запускает расчёт из UI.
2. Для части строк появляется **итоговая цена** и **расшифровка** (формула + компоненты + источники).
3. Для строк без возможности расчёта виден **статус и причина**, цена не подставляется «из воздуха».
4. Ошибка одной строки не валит весь прогон.
5. На экране есть **сводные метрики**: доля успеха, счётчики невозможности расчёта и ошибок (нет формулы, нет котировки/курса, ошибка выражения и т.д.).
6. При нескольких формулах кандидаты видны; default — самая поздняя дата создания.

## 9. Риски и упрощения MVP

| Риск | Упрощение на День 3 |
| --- | --- |
| Сложный синтаксис формул Pricing | только `+ - * / ()`; иначе `UNSUPPORTED_EXPRESSION` |
| Правила периода котировки/курса (`YQ100`, `YC102`…) | дата = `period` строки или календарный месяц; помечать в расшифровке |
| Неполное покрытие `material_groups.hname` vs формулы | явный статус / warning, не тихий skip |
| H2 in-memory | данные и результаты живут, пока жив процесс; для демо не перезапускать без повторной загрузки |
| Эталон ±3% | метрика в UI-заготовке; заполнение — когда будет эталон |
