# Шаг 1: Массовый импорт файлов (Bulk Upload), Liquibase и UI отображения

## Стек
Java 17, Spring Boot 3, Spring Data JPA, База данных H2, Liquibase (для миграций БД), Thymeleaf + HTMX + Tailwind CSS.

## Задача
Реализовать единую зону загрузки для нескольких файлов одновременно (`MultipartFile[]`). Бэкенд должен автоматически определять целевую таблицу H2 по имени файла, валидировать данные и сохранять их. Таблицы в БД создаются строго через Liquibase без физических ограничений Foreign Key. В UI отображаются две главные вкладки: "Прогноз спроса" и "Каталог формул".

## Концепция интерфейса (UI)
Экран разделен на две зоны:
1. **Зона загрузки (Сверху):**
   * Единый `Drag-and-Drop` контейнер, поддерживающий множественный выбор (`multiple`).
   * Кнопка "Загрузить все".
   * Ниже — интерактивный лог обработки. Например:
      * `📄 sales_forecast_2026.xlsx ➔ Успешно импортировано в таблицу Прогноз продаж (3076 строк)`
      * `❌ quotes.xlsx ➔ Ошибка: [текст ошибки]`
2. **Зона результатов (Снизу — две вкладки):**
   * **Вкладка 1:** "Прогноз спроса" (Данные из таблицы Прогноза в H2).
   * **Вкладка 2:** "Каталог формул" (Данные из таблицы Формул в H2).

## Управление структурой БД (Liquibase)
Cursor, настройка базы данных должна выполняться по следующим правилам:
1. Отключи автоматическую генерацию таблиц Hibernate в `application.properties`: `spring.jpa.hibernate.ddl-auto=none`.
2. Включи Liquibase и создай файл миграции `db/changelog/db.changelog-master.yaml`.
3. Напиши скрипты создания всех 8 таблиц на основе структуры из `@data-model.md`.
4. **СТРОГОЕ ПРАВИЛО:** Каждая таблица должна иметь суррогатный первичный ключ `id` (автоинкремент / identity). Все остальные бизнес-ключи и идентификаторы (такие как `formula_key`, `customer_asv`, `code_nsi`, `quote_code`) создавай как обычные текстовые колонки (`VARCHAR`).
5. **ЗАПРЕЩЕНО:** Использовать ограничения `FOREIGN KEY` и `UNIQUE` на уровне Liquibase-скриптов. База данных должна принимать данные «как есть», без проверок ссылочной целостности, чтобы избежать ошибок при неполных справочниках и нарушении очередности загрузки.

## Алгоритм маппинга и парсинга на Бэкенде
1. **Сопоставление по имени:** Бэкенд анализирует оригинальное имя файла (`file.getOriginalFilename().toLowerCase()`) и сопоставляет его с JPA-сущностями:
   * `*ssp*` ➔ Таблица `DemandForecast`
   * `*formula_components*` ➔ Таблица `FormulaComponent`
   * `*formulas*` (но не components) ➔ Таблица `Formula`
   * `*term_types*` ➔ Таблица `TermType`
   * `*material_groups*` ➔ Таблица `MaterialGroup`
   * `*quote_mapping*` ➔ Таблица `QuoteMapping`
   * `*quotes*` (но не mapping) ➔ Таблица `QuoteValue`
   * `*currency_rates*` ➔ Таблица `CurrencyRate`
2. **Строгая последовательность:** При получении массива файлов `MultipartFile[]` бэкенд сортирует их в памяти по приоритету импорта:
   * Приоритет 1: `term_types`, `material_groups`
   * Приоритет 2: `formulas`, `quote_mapping`
   * Приоритет 3: `formula_components`, `quotes`
   * Приоритет 4: `currency_rates`
   * Приоритет 5: `ssp`
3. **Очистка данных:** Перед парсингом конкретного файла очищается только его целевая таблица через Spring Data JPA репозиторий (`repository.deleteAll()`).
4. **Дедупликация котировок:** Для файла `quotes.xlsx` перед сохранением в БД отфильтруй дубликаты по бизнес-ключу (`quote_code`, `quote_date`, `quote_type`). Если они совпадают, оставляй только одну строчку с максимальным (самым поздним) значением `tech_load_ts`.

## Правила для JPA-сущностей в Java
* В классах сущностей (`@Entity`) используй аннотации `@Column(name = "...")` для маппинга на колонки Liquibase.
* Связи между сущностями (например, компонента с формулой) делай через обычные поля `String formulaKey` или `String quoteCode`. **Не используй** аннотации `@ManyToOne`, `@OneToMany` или `@JoinColumn`.
